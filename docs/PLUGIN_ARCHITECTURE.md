# プラグイン受け皿仕様

## 1. 目的

本アプリへGmail、YouTube、RSS、ニュース、SNS、カレンダー、ローカル操作、AI会話などの機能を追加する際、個別サービスの処理を画面やキャラクター制御へ直接埋め込まないための共通基盤を定義します。

この仕様でいう「プラグイン」は、外部サービスまたはローカル機能から情報・操作・会話材料を提供するモジュールです。

## 2. 設計原則

- UI、キャラクター、外部サービスを疎結合にします。
- 全プラグインは共通のイベント形式へ変換します。
- APIキー、OAuthトークン、Cookieなどの秘密情報をイベント本文へ含めません。
- 取得処理と表示判断を分離します。
- 通知の重複、過剰通知、深夜通知をホスト側で抑制します。
- 重要な情報はキャラクターAだけで完結させ、Bは補足・相づち・注意喚起を担当します。
- API未接続時でもローカル機能と定型会話は動作可能にします。

## 3. 初期対応方式

### 3.1 内蔵プラグイン

Swift Package内へコンパイルして同梱する方式です。初期版はこの方式を正式対応とします。

対象例:

- OpenAI会話
- Gmail新着確認
- YouTube配信通知
- RSS・Googleニュース
- macOSアプリ起動
- ローカル時報・記念日

### 3.2 外部プラグイン

任意の第三者コードをアプリ本体へ直接ロードする方式は採用しません。macOSでは署名、権限、秘密情報、クラッシュ波及の問題があるためです。

将来はXPC Serviceまたは別プロセスを用い、JSONメッセージで通信する隔離方式を検討します。外部プラグインへ渡す権限はマニフェストで宣言し、ユーザー承認を必須とします。

## 4. 共通データモデル

### 4.1 PluginManifest

各プラグインが自己申告する情報です。

- `id`: 逆DNS形式の一意ID
- `name`: 表示名
- `version`: セマンティックバージョン
- `capabilities`: 提供機能
- `requiredPermissions`: 必要権限
- `defaultPollingInterval`: 標準確認間隔

### 4.2 PluginCapability

初期定義:

- `eventSource`: 新着情報を取得します
- `command`: ユーザー操作を実行します
- `conversationContext`: AI会話へ材料を渡します
- `notificationAction`: 通知から既読化・開く等を実行します
- `settings`: 個別設定画面を提供します

### 4.3 CompanionEvent

すべての情報源は、最終的に次の共通イベントへ変換します。

- `id`: サービス側IDを含む安定した識別子
- `pluginID`: 発生元プラグイン
- `kind`: mail / social / video / live / news / calendar / system / conversation
- `title`: 一行見出し
- `body`: キャラクターが読み上げ可能な本文
- `sourceName`: Gmail、YouTube等
- `sourceURL`: 開く先
- `occurredAt`: 発生日時
- `priority`: low / normal / high / critical
- `deduplicationKey`: 重複排除キー
- `suggestedSpeaker`: A / B / automatic
- `metadata`: 秘密情報を含まない追加情報
- `actions`: 開く、既読、後で通知等

## 5. プラグインの種類

### EventSourcePlugin

定期確認または手動更新でイベントを返します。

例:

- Gmail未読メール
- YouTube配信開始
- RSS新着記事
- カレンダー予定

### CommandPlugin

ユーザー入力や通知ボタンから処理を実行します。

例:

- Safariを開く
- メールを既読にする
- YouTube配信ページを開く
- ChatGPTアプリを起動して入力文をクリップボードへコピーする

### ConversationContextPlugin

AIへ送る前に、許可された範囲の追加情報を提供します。メール本文など機微情報は、プラグイン設定とユーザー操作の両方で明示的に許可された場合だけ渡します。

## 6. ホスト側の処理順

1. `PluginRegistry`が有効なプラグインを保持します。
2. `PluginScheduler`が各プラグインの確認時刻を管理します。
3. プラグインが`CompanionEvent`を返します。
4. `EventPipeline`が重複排除、時間帯、優先度、ミュート条件を評価します。
5. `ConversationCoordinator`がA/Bの発言へ変換します。
6. `NotificationCenter`またはデスクトップ吹き出しへ表示します。
7. イベントIDと処理結果をローカル保存します。

外部サービスの取得失敗はキャラクター会話を停止させず、プラグイン単位で隔離します。

## 7. 発言ルール

### キャラクターA

- 質問への主要回答
- メール・ニュース・予定の要約
- エラー内容と必要な操作
- ユーザー判断が必要な情報

### キャラクターB

- 短い補足
- 相づち
- 締切や重要度への注意喚起
- Aの回答を妨げない軽い掛け合い

`priority == critical`のイベントはAのみが簡潔に通知します。Bの発言生成を待ちません。

## 8. 認証と秘密情報

- APIキーとOAuthトークンはmacOS Keychainへ保存します。
- `UserDefaults`、ログ、イベント履歴、AIプロンプトへ秘密情報を保存しません。
- OAuthはAuthorization Code + PKCEを基本とします。
- Gmail等の権限は最小スコープにします。
- XはAPI制約・料金・仕様変更が多いため、公式APIまたはユーザー指定RSS等の正規手段だけを利用します。画面スクレイピングを標準機能にしません。

## 9. 通知制御

ホスト側で以下を共通実装します。

- 同一`deduplicationKey`の再通知抑止
- プラグインごとの最短確認間隔
- 1時間・1日あたりの通知上限
- 集中モード、画面ロック、フルスクリーン中の抑止
- 静音時間帯
- low優先度イベントのまとめ通知
- high以上の即時通知

## 10. 障害対策

- プラグインごとにタイムアウトを設定します。
- 連続失敗時は指数バックオフします。
- 失敗は他プラグインへ波及させません。
- 最終成功日時、最終エラー、次回確認日時を設定画面へ表示します。
- 不正なイベントはホスト側で拒否します。

## 11. 初期実装範囲

このブランチでは以下を追加します。

- 共通マニフェスト
- 共通イベント
-権限・能力・優先度・発言者の型
- EventSource / Command / ConversationContextプロトコル
- 型消去されたプラグイン登録機構
- 重複排除と簡易通知判断を行うイベントパイプライン
- モックプラグインを作成できるテスト基盤

実サービスのOAuth接続、設定UI、バックグラウンドスケジューラ、XPC外部プラグインは次段階とします。

## 12. 推奨実装順

1. プラグイン一覧・有効無効設定
2. RSSプラグインで取得・重複排除・通知を実証
3. YouTube Data API
4. Gmail OAuth
5. OpenAIによるイベント要約とA/B掛け合い
6. カレンダー
7. 外部プラグイン用XPC仕様

RSSを最初にする理由は、OAuthや高額APIなしで基盤全体を検証できるためです。
